# Benefit Before Feature Patterns

## Role in LP

機能より先にメリットを伝え、ユーザーが「自分への効果」を最初に理解できるようにする。

## Best Use Cases

- 解決策セクション
- 機能紹介セクション
- FV のサブコピー

## Required Elements

- ベネフィットを最初に提示
- 機能はベネフィットの裏付けとして
- 数値根拠
- 一人称表現

## High Conversion Pattern

### 型A: ベネフィット → 機能の説明型

```markdown
## 採用工数を月60%削減します（自社調べ・100社平均）

その理由は、3つの自動化機能：

### 1. 応募者の自動振り分け
求人媒体から応募が来ると、自動で一覧表に整理されます。
スプレッドシート管理は不要です。

### 2. 選考結果の自動返信メール
合否判断後、テンプレートメールが自動送信されます。
対応漏れがゼロになります。

### 3. ダッシュボードでリアルタイム可視化
採用状況・経路別CV率がリアルタイムで分かります。
経営層への報告は、画面を見せるだけです。
```

### 型B: あなたの業務がこう変わる型

```markdown
## あなたの業務が、こう変わります

### 朝
ダッシュボードを開くだけ
「あ、ここ対応済みだな」と確認するだけ

### 日中
重要な意思決定に集中
手作業のデータ集計はゼロ

### 夕方
自動レポートをチェック
残業せずに帰れる

これを実現するのは、API連携・自動メール・ダッシュボード機能です。
```

### 型C: Before / After 型

```markdown
## 導入前と導入後

| 業務 | 導入前 | 導入後 |
|---|---|---|
| 応募者管理 | スプレッドシート手動入力 | 自動振り分け |
| 対応メール | 1通ずつ手作業 | テンプレート自動送信 |
| 経営報告 | 週8時間の手作業 | 自動レポート（実質ゼロ） |

工数: 月150時間 → 月60時間（60%削減）

※ 自社調べ・100社平均
```

## Bad Pattern

### NG型: 機能羅列のみ

```
## 主要機能

- API連携
- 自動メール
- ダッシュボード
- アクセス権限管理
- データエクスポート
- レポート機能
```

→ 「だから何？」となり離脱。

### NG型: 抽象的すぎるベネフィット

```
## 採用業務を効率化します

私たちのサービスは、あなたの採用業務を変えます。
```

→ 具体性なく信頼されない。

## Copy Examples

### 良い例
- 「月100時間の業務を、月40時間に減らします（自社調べ）」
- 「上司から『楽になったね』と言われる業務改善」
- 「あなたが本来やるべき仕事に、時間を使えます」

### 悪い例
- 「画期的な業務効率化」（抽象）
- 「最高の機能」（根拠なし）
- 「業界初」（業界差別化が分からない）

## Design Notes

- ベネフィットは大きく
- 機能は補足として小さく
- アイコンやスクリーンショットでビジュアル補助
- 数字は太字 or バッジ

## Measurement Notes

- セクション閲覧時間
- ベネフィット部分のスクロール深度

## Checklist

- [ ] ベネフィット → 機能 の順
- [ ] 数値根拠（出典付き）
- [ ] 一人称表現
- [ ] 「上司・経営層」など周囲の反応も含む
- [ ] Before / After を時系列で
