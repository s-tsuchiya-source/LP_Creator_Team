# escalation-rule.md — エスカレーション規程

ディレクター単独で判断困難な場合、追加エージェントへエスカレーションする条件。

## エスカレーション要否判定

| シナリオ | 必須エージェント |
|---|---|
| 業界が法規制対象（医療 / 美容 / 健康食品 / 金融 / 不動産） | `lp-legal-expression-checker` |
| 個人情報入力フォームあり | `lp-security-checker` |
| 計測要件が複雑（マルチドメイン / Server-side） | `lp-ga4-gtm-specialist` + `lp-conversion-tracking-specialist` |
| CRM/MA連携要 | `lp-crm-ma-integration-specialist` |
| 既存LP改善で行動データあり | `lp-heatmap-analyst` |
| BtoB高単価で商談化率重要 | `lp-business-model-analyst` |
| 採用LPで求職者心理重要 | `lp-customer-insight-researcher` |
| 高単価ブランド・ブランド毀損リスク | `lp-brand-reviewer` |
| 経営判断が必要（事業整合 / KPIインパクト大） | `lp-executive-reviewer` |
| GitHub公開前構成レビュー | ディレクターが構成レビュー |

## ユーザーへのエスカレーション条件

以下はディレクターが単独で進めず、ユーザーに確認する：

- CV地点が確定していない
- ターゲット定義が曖昧（複数想定）
- 商材の効果効能が法令該当の可能性
- 実績・数字の根拠が不明
- ブランドガイドラインが見つからない
- 公開期日が不明
- 出稿媒体（広告 / SEO）が不明

## エスカレーション時の指示テンプレ

```
[エージェント名] にエスカレーション:
- 理由: [なぜディレクター単独で判断できないか]
- 確認したい論点: [具体的問い]
- 期待回答形式: [PASS / NEEDS WORK / BLOCK + 理由]
- タイムボックス: [この時間内で]
```

## 並行エスカレーション

複数エージェントへ並行依頼可能なケース：

- 法務 + セキュリティ（独立観点）
- 経営 + ブランド + CV観点（最終レビュー）
- ペルソナ + 競合 + 市場（リサーチ並行）

## エスカレーション後の処理

1. 各エージェントから結果を受領
2. 矛盾があれば調停（最も保守的な判断を優先）
3. 必要に応じてユーザーに確認
4. 最終承認に向けて統合
