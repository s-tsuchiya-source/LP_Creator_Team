# メリット実感設計

ユーザーが LP を読みながら「自分にとってのメリット」を直感的に実感できるよう、機能ではなくベネフィットを軸に設計します。

## 機能 → ベネフィット 変換の基本

### 変換式

```
[機能] → [何が楽になるか] → [何が減るか] → [何が増えるか] → [誰に評価されるか] → [どんな未来が得られるか]
```

### 例

| 機能 | ベネフィット |
|---|---|
| API連携 | 既存ツールとの二重入力をなくし、月X時間削減 |
| 自動メール返信 | 応募者対応の遅延ゼロ・対応漏れの不安から解放 |
| ダッシュボード | 経営層への報告時間がゼロ、上司から「分かりやすい」と評価 |
| 24時間サポート | トラブル時にすぐ相談できる安心感 |
| 業界特化テンプレート | 設定不要で、最短2週間で導入完了 |

## メリットの3層構造

### 1. 短期メリット（導入直後）

- 申込の手軽さ
- 即時の利便性
- すぐに体験できる効果

### 2. 中期メリット（1〜3ヶ月後）

- 業務効率の実感
- 数値改善（工数 / コスト）
- チーム内の評価向上

### 3. 長期メリット（半年〜1年）

- 組織レベルの変化
- 経営指標への影響
- 業界内ポジション向上

LP内で**3層全てに触れる**ことで、ユーザーは「短期だけでなく、長期も期待できる」と感じます。

## メリットのカテゴリ別整理

### 経済的メリット
- コスト削減
- 売上向上
- ROI / 投資回収期間
- 工数削減 → 人件費換算

### 時間的メリット
- 業務時間削減
- 意思決定スピード向上
- 待ち時間ゼロ

### 心理的メリット
- 不安からの解放
- 失敗回避
- 安心感
- 達成感

### 業務上のメリット
- 抜け漏れ防止
- データ可視化
- 属人化解消
- 業務の見える化

### 社内評価・説明上のメリット
- 上司に推せる
- 経営層への報告が楽
- 部門間調整が減る
- 稟議が通りやすい

### 失敗回避のメリット
- セキュリティ事故防止
- データ消失防止
- コンプライアンス遵守
- ブランド毀損回避

## ターゲット別メリット例

### BtoB SaaS
```
短期: 申込3分 / 14日間無料
中期: 業務工数 月50時間削減
長期: 組織のデータ可視化、経営判断スピード向上
心理: 「もう手作業で疲弊しない」
社内: 「経営層への報告が楽になった」
```

### 採用LP
```
短期: カジュアル面談で気軽に話せる
中期: スキルが活きる業務で成長実感
長期: キャリアアップ・市場価値向上
心理: 「ここなら長く働けそう」
社内: 「メンバーと働きたい」
```

### SaaS（中小企業）
```
短期: 14日間無料・クレカ不要
中期: チーム連携の改善
長期: 業務全体の効率化
心理: 「もう情報が散らばらない」
社内: 「現場が定着しそう」
```

### 広告流入LP
```
短期: 3分で資料DL / 即日対応
中期: 業界共通課題の解消
長期: 競合との差別化
心理: 「広告で見たメッセージそのままだ」
社内: 「上司に推せる根拠資料がある」
```

### EC / 物販
```
短期: 30%OFF + 送料無料
中期: 30日間使って効果実感
長期: 習慣化で生活が変わる
心理: 「失敗しても返金保証」
社内: -
```

## Before / After の見せ方

### NG（誇大）
```
Before: 営業辞めたい
After: 売上3倍！
```

### OK（具体的）
```
Before: 月150時間の採用業務、手作業でミス頻発
After: 月60時間まで削減、ダッシュボードで可視化
※ 自社調べ・導入企業100社平均（2026年4月時点）
```

## メリット訴求の配置

LP内で以下の位置にメリットを配置します。

| セクション | メリットの種類 |
|---|---|
| FV | 主要ベネフィット 1つ（最も刺さるもの） |
| 課題訴求 | メリット欠如時の不利益（暗示） |
| 解決策 | 機能 × ベネフィット 3〜5個 |
| 強み | 競合との差別化メリット |
| 実績 | 数値メリット（出典付き） |
| 事例 | 具体的なベネフィット実現ストーリー |
| 価格 | コスト対効果（ROI） |
| 最終CTA | 主要ベネフィット 再訴求 + 行動メリット |
