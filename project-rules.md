# project-rules.md — プロジェクト運用ルール

## 1. 役割の階層

```
ユーザー
  ↓
lp-director（統括）
  ↓
専門エージェント群（戦略 / リサーチ / IA / コピー / デザイン / 実装 / 計測 / QA）
  ↓
レビューボード（経営 / CV / ブランド / 最終承認）
```

## 2. 制作工程の標準フロー

1. **Brief**（ヒアリング・前提整理）
2. **Strategy**（戦略・ペルソナ・訴求軸）
3. **Structure**（セクション構成・ワイヤーフレーム）
4. **Copy**（コピー全文）
5. **Design**（デザイン方針・ビジュアル指示）
6. **Code**（HTML / CSS / JS 実装）
7. **Measurement**（GA4 / GTM / CV計測）
8. **QA**（表示・リンク・フォーム・法務・速度）
9. **Review**（経営 / CV / ブランド）
10. **Release**（最終承認）

各フェーズに対応する品質ゲートを `quality-gates/` に定義。

## 3. エージェント呼び出しルール

- **ディレクター経由のみ**: ユーザーは原則 `lp-director` のみを起点とする
- **必要最小限**: ディレクターは案件タイプに応じて呼び出すエージェントを最小化
- **目的明示**: ディレクターは各エージェント呼び出し時に「目的・入力・期待出力」を明示
- **重複禁止**: 同一観点を複数エージェントで重複させない

## 4. 成果物の保存先

| フェーズ | 保存先 |
|---|---|
| Brief | `outputs/01_brief/` |
| Strategy | `outputs/02_strategy/` |
| Structure | `outputs/03_structure/` |
| Copy | `outputs/04_copy/` |
| Design | `outputs/05_design/` |
| Code | `outputs/06_code/` |
| Measurement | `outputs/07_measurement/` |
| Review | `outputs/08_review/` |

ファイル命名規則: `YYYY-MM-DD_案件名_フェーズ.md`

## 5. 命名規則

- エージェント名: 英小文字 + 数字 + ハイフン（例: `lp-director`）
- ファイル名: 小文字 + ハイフン or アンダースコア
- 案件ID: 必要に応じてユーザーが指定（例: `lp-2026-04-001`）

## 6. 品質基準

- 戦略: ターゲット・訴求軸・差別化が明文化されている
- 構成: FV / 問題 / 解決 / 証拠 / FAQ / CTA を必須含有
- コピー: 誰向け・何が得られる・なぜ信頼できる・なぜ今 が明確
- デザイン: 視線誘導・CTA可視性・SP対応
- 実装: セマンティックHTML / レスポンシブ / 軽量
- 計測: CTAクリック / フォーム到達 / 開始 / 送信 / サンクスページ
- QA: 表示 / リンク / フォーム / 速度 / 法務 / a11y

## 7. 禁止事項

- 全エージェント一括起動
- 検証なし数値の使用
- No.1表現・最高表現の根拠なし使用
- 秘密情報・API キーの記載
- 実顧客情報の examples 混入
- 品質ゲートのスキップ

## 8. ナレッジ追加・改善

新しいパターン・知見が得られた場合：

1. `knowledge/` の該当ディレクトリに追加（例: `knowledge/copy-frameworks/`）
2. 既存ファイルとの重複を回避
3. 必要に応じてエージェントの参照を追加
4. 大きな変更は `tasks/active-task.md` に記録

## 9. セッション節約

- ディレクターは先に仮説を立て、エージェント数を最小化
- 中間成果物の冗長化を避ける
- 同一情報の繰り返し参照を避ける
- 重要不明点のみ確認、それ以外は仮置き補完
